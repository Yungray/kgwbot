# Review Quality — 산출물 횡단 품질 검토

여러 전문가의 산출물을 받아 일관성·완결성·회귀 위험을 점검한다.

## 작업 절차

### 1. 검토 범위·기준선 확보
- 어떤 산출물을 검토하는지 (PRD / 디자인 / 프롬프트 / 코드 / 통합)
- 기준선: 기존 동작, `~/AGENTS.md` 디자인 토큰, 이전 결정 사항

### 2. 항목별 점검

#### 일관성
- 기획 AC ↔ 디자인 컴포넌트 ↔ 프롬프트 출력 ↔ 코드 동작이 일치하는가
- 용어 일관성: "모듈" vs "그룹", "답변 가이드" vs "응대 가이드" 등 표기 통일
- 디자인 토큰 일관 사용

#### 완결성
- User Story의 모든 AC가 구현/디자인되었는가
- 빈 상태·로딩·에러·0건 결과·네트워크 실패 시나리오 모두 정의
- 키보드만으로도 사용 가능한가

#### 모듈 독립성 (이 프로젝트 특수)
- 전자결재/인사 시스템 어디서든 동일 기능이 동일 동작
- 세션·메시지·토큰·group_id가 모듈별로 분리됨
- pinned_group enforcement가 모든 도구에 일관 적용됨

#### 엣지 케이스
- 토큰 미설정/만료, group_id 권한 누락
- IME 입력 두 번 전송, 빠른 모듈 전환, 백그라운드 응답 도착
- LLM 응답 빈/포맷 깨짐, 도구 호출 실패

#### 회귀 위험
- 변경이 기존 기능에 미치는 영향
- 데이터 마이그레이션 필요 여부 (예: localStorage 키 변경)

### 3. 결과 분류·기록
각 발견 사항을 다음 4단계로:
- ✅ 잘 됨
- ⚠️ 개선 권장 (Should-fix)
- ❌ Must-fix
- 🔄 회귀 위험

### 4. 후속 조치 권고
- 어떤 에이전트에게 무엇을 의뢰할지 명시
  - "AC #3 추가는 product-planner에게"
  - "에러 상태 디자인은 ux-ui-designer에게"
  - "프롬프트의 출력 포맷 강제는 prompt-engineer에게"

## 출력 규칙

### 리포트 템플릿
```
## 검토 대상: [이름]
- 검토일: YYYY-MM-DD
- 검토자: qa-reviewer

### ✅ 잘 된 점
- …

### ⚠️ 개선 권장
- [위치] [현상] → 기대값 → 영향도 → 권장 조치
- …

### ❌ Must-fix
- [위치] 재현 단계 1) … 2) … / 차단 사항

### 🔄 회귀 위험
- [기존 기능 X] 영향: 어떤 시나리오에서 깨질 수 있음 / 점검 포인트

### 후속 의뢰
- product-planner: …
- ux-ui-designer: …
- prompt-engineer: …
```

## 점검 체크리스트 (이 프로젝트 특수)

- [ ] 모듈 독립성 (전자결재/인사 동일 동작)
- [ ] pinned_group 강제 (LLM이 우회 못함)
- [ ] per_id_counts 노출 (multi-group 검증)
- [ ] 한국어 IME 입력 처리
- [ ] 모듈별 세션·메시지 분리
- [ ] 라이트 LNB + Indigo 토큰 일관
- [ ] 환영 카드 ↔ 메시지 상태 전환 자연스러움
