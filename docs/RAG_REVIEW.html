지금 구조의 본질과 전환의 의미

현재 시스템은 데이터를 소유하지 않습니다. 매 질의마다 그룹 토큰으로 api.agit.in/v2/search(substring exact 매칭)를 실시간 호출해 결과만 중계합니다.

SEARCH_SPEC B-4의 핵심 한계: "Agit API 인덱싱은 서드파티 영역 — 우리가 손댈 수 있는 건 쿼리 전략(정렬·동의어·기간·페이징)뿐." 벡터/Graph RAG로 간다는 건 이 전제를 "우리가 데이터 사본 + 인덱스를 소유한다"로 바꾸는 것이며, 검토의 거의 모든 항목이 여기서 파생됩니다.

✅ 좋은 소식: get_group_task_stats가 이미 search.total을 page=1부터 has_more=false까지 전수 순회하고 있습니다(agit.py:1077–1116). 즉 Agit에서 글을 벌크로 빼올 수 있는 경로가 이미 증명되어 있어, 자체 인덱스 구축의 첫 관문은 통과한 상태입니다.

1

검토 단계(Phase 0) — 빌드 전에 닫을 결정 게이트

우선순위 순으로, 빌드보다 먼저 답해야 하는 질문들입니다.

①

데이터 접근 · 동기화 Feasibility

벌크 추출 경로는 확인됨. 검증 필요: 댓글 포함 여부(처리상태는 root 글이 아니라 댓글 스레드에 있음 → 글마다 fetch_thread_detail 필요 → 초기 적재 시 콜 폭증·rate limit), 증분 동기화(웹훅 없으면 날짜 윈도우 폴링 + 댓글이 나중에 달리면 스레드 재색인 트리거).

②

데이터 프로파일링

그룹별 글 수 · 증가율 · 댓글 밀도 · 평균 본문 길이. → 벡터 DB 선택과 재색인 주기를 결정. 현재 규모면 대형 분산 스토어는 과합니다.

③

권한 · 격리 모델 보존 보안 필수

지금은 그룹별 토큰 + PINNED_GROUP/PINNED_MODE로 그룹 간 결과 누수를 차단(report 모드는 voc_report_ids만). 벡터 인덱스에서도 깨지면 안 됨 → 모든 청크에 group_id 메타데이터를 박고, 검색 시 pinned 그룹/모드로 메타데이터 필터링을 강제.

④

PII · 정보보호 검토 민감 데이터

대상이 사내 인사·전자결재 데이터. 임베딩해 저장하는 순간 민감 사내 데이터의 파생 사본이 생김. 외부 임베딩 API를 쓰면 본문이 제3자로 전송됨 → 자체 호스팅 임베딩으로 회피 가능. (자세한 배경은 §5)

⑤

평가 하네스 먼저 튜닝의 전제

정답셋(쿼리→기대 글) 기반 recall@k / precision@k로 현재 키워드 시스템의 베이스라인부터 측정. 대표 실패 케이스 "2022 마이너스 정산"을 회귀 앵커로. 숫자 없으면 "벡터가 낫다"는 신앙이 됩니다.

검토 결론 형태: ①~⑤를 통과하면 게이트 OK → Phase 1로. ④에서 막히면 자체 호스팅 임베딩으로 설계 제약을 먼저 확정한 뒤 진행.

벡터 검색에 필요한 작업 (Phase 1 — Hybrid MVP)

단계작업권장 / 주의
적재search.total 페이지 순회로 글 수집 → 글마다 fetch_thread_detail로 댓글까지 → "글+댓글" 단위 정규화·청킹기존 _classify_resolution · 메타데이터를 청크에 함께 저장
임베딩한국어 구어↔공식어 매칭이 목적 → 다국어 품질 중요BGE-m3 / multilingual-e5(자체 호스팅 → PII 회피) vs Gemini embedding(운영 단순하나 외부 전송)
벡터 스토어group_id 메타 필터링 + 소규모pgvector(세션 SQLite/Redis 백로그와 DB 통합) 또는 Qdrant. 대형 분산(Milvus)은 과함
검색하이브리드 — 기존 키워드(BM25/substring) + 벡터를 RRF로 융합순수 벡터로 갈아엎지 말 것. ABC-123·문서번호는 임베딩이 약하고 키워드가 강함
재랭킹기존 _case_quality_score(최신성·댓글 수·처리상태)는 그대로 유지, 검색 위에 얹음선택적으로 cross-encoder(bge-reranker) 추가
통합find_similar_cases를 자체 인덱스 하이브리드 검색으로 교체, feature flag 뒤에서fetch_thread_detail은 신선도 위해 계속 Agit 라이브 호출
설계 원칙 2가지 — ① 기존 키워드 파이프라인(동의어·조사 정규화·고신호 패턴)을 버리지 말고 하이브리드로 흡수. ② 검색(retrieval)과 비즈니스 재랭킹(처리상태/최신성)을 분리 — 이미 코드가 이 구조라 교체가 쉽습니다.
3

Graph RAG — 솔직한 평가

직관은 맞는 부분이 있습니다. 다만 "바로 Graph RAG부터"는 권하지 않습니다.

✓ 실제로 빛나는 곳

  • 처리 계보: "이 유형은 보통 어떻게 처리됐나" → 이슈→처리방식→재사용 답변 다홉 탐색 (이미 처리상태 분류 중이라 엣지로 자연 연결)
  • 문서 간 교차 추론: 정산서 이슈 ↔ 관련 결재선/전표 이슈 — 순수 벡터가 놓치는 부분
  • 엔티티 집계: "LDAP 권한 관련 모든 문의"를 그룹 넘어 엔티티 기준으로

✕ 과한 / 위험한 부분

  • 그래프 구축 비용: 한국어 자유 텍스트에서 LLM 엔티티 추출 → 전체 코퍼스 + 증분마다. 비용·오류·유지보수 부담
  • 핵심 유스케이스는 벡터로 80% 커버: "유사 사례로 답변 초안"은 포인트 조회 → 하이브리드 + 메타 필터로 대부분 해결
  • 집계 수요는 이미 일부 충족: VOC 리포트 모드의 per_group_counts·get_group_task_stats
권고 — 단계적 + "GraphRAG-lite": 그래프는 나중에, 자유 텍스트 LLM 추출이 아니라 이미 뽑고 있는 구조화 메타(문서유형·시스템·처리상태) 위에 가벼운 "메타데이터 그래프"로 올리면 구축비용 없이 multi-hop·엔티티 집계를 얻습니다. 풀 Graph RAG(community detection·global summary)는 집계형 질문이 실제 요구사항으로 올라올 때만.

부록 — PII/컴플라이언스 검토는 어떤 배경에서?

외부에서 받은 요구사항이 아니라, 코드베이스의 신호로부터 추론한 리스크 플래그입니다. 근거 4가지:

① 데이터 출처가 사내 시스템
시스템 프롬프트: "카카오/디케이테크인 사내 아지트 데이터" — 공개 데이터가 아님

② 다루는 그룹이 인사·전자결재
AGIT_TOKEN_ELEC(전자결재) · AGIT_TOKEN_HR(인사 시스템) — 민감정보 대표 카테고리

③ 문서 유형이 개인·재무 정보
정산서·법인카드·전표·휴가/연차·발령·결재선 — 급여·정산·인사 기록

④ 시스템이 "격리"를 보안요건으로 취급
PINNED_GROUP 강제 override·그룹별 토큰 분리 = 접근통제 대상임을 설계가 전제

왜 "벡터 RAG로 가면" 새로 문제가 되나

현재 (stateless 프록시)벡터 RAG
데이터 보유안 들고 있음 — 결과만 중계파생 사본을 영구 저장
접근통제 위치Agit가 토큰 단위로 강제우리 벡터 DB가 직접 책임
외부 전송없음 (검색은 Agit 내부)외부 임베딩 API 쓰면 본문이 제3자로 전송
이건 차단 사유가 아니라 "설계 제약을 미리 확정하라"는 신호입니다. 제가 법적 의무 문서를 본 게 아니라 단정이 아닌 플래그이며, 실제 적용 강도는 사내 정책에 달렸습니다. 다만 임베딩 모델 선택(자체 호스팅 vs 외부)·저장소 위치 같은 번복하기 비싼 결정이 흔들리지 않도록 Phase 0에서 짚는 게 맞습니다. 데이터 민감도가 낮거나 사내 정책이 이미 정해져 있다면 가볍게 넘겨도 됩니다.
